Abstract

ResourceFlow is a donation and resource allocation platform that connects aid donors with people in need across Ghana. The system handles monetary and in-kind donations, request management, vulnerability-based prioritization, NGO matchmaking (Find NGOs), logistics tracking, and last-mile delivery verification. This technical document provides an in-depth examination of the system architecture, technology stack, database design with full entity relationships and cardinalities, deployment infrastructure, and implementation details.

The platform is built with a Laravel API backend, React SPA frontend, PostgreSQL database, Paystack payment integration, QoreID Ghana Card verification (Verify by ID number and image-based workflows), and react-webcam for document and selfie capture. It supports multiple donor types including Angel Donors (no ID verification required, donations capped at GH₵5,000). Users can skip verification during registration (Ghana Card and NGO RG document not required); they are created with is_verified=false and redirected to /verification-wait to complete later. ProtectedRoute and RoleProtectedRoute enforce verification and role-based access. The document includes detailed code explanations for the SVG heat map, MVC implementation, Paystack integration, QoreID Ghana Card verification (both ID lookup and image verification with workflow fallback), Find NGOs matchmaking, camera capture for delivery verification, and the updated delivery proof workflow (Ghana Card photo, recipient photo, recipient Ghana Card number verification—no signature). Deployed on Hostinger VPS or Laravel Cloud with Nginx, SSL, and automated deployment scripts. Deployment challenges (Git ownership, CORS, camera/HTTPS, divergent branches) are documented in section 6A.4.

Chapter One: Introduction

1.1 Problem Statement

Aid distribution systems often struggle with fragmented workflows, poor transparency, and inefficient matching between donors and recipients. Donors have difficulty understanding where need is greatest. Recipients wait without visibility into allocation decisions. Administrators lack tools to prioritize fairly and track delivery end to end. Fraud and self-dealing remain concerns when the same party can donate and receive.

ResourceFlow tackles these issues by offering transparency through audit trails and verification workflows. Every allocation, verification, and delivery action is logged. Prioritization uses vulnerability scores and urgency calculations so that the most critical requests receive attention first. Traceability runs from donation to delivery with biometric verification at handover. Compliance is enforced through Ghana Card verification, NGO vetting, and donor tax rules such as the 10 percent assessable income cap for corporate donations and the GHS 10,000 anonymous limit for individuals. Fraud mitigation includes self-dealing guards that check phone numbers, Ghana Card, and account IDs to prevent donors from allocating to themselves.

Beyond these core challenges, the system must also handle the complexity of multiple donor types (individual and corporate), multiple request types (goods, monetary, services), and the need for partial funding through contributions. The logistics layer must support real-time driver tracking across Ghana's regions and capture delivery proof at the point of handover to close the audit loop. All of these requirements have been addressed in the deployments handled for this project.

1.2 Objectives of the Software

The software aims to connect donors and recipients by enabling monetary and in-kind donations to verified aid requests. It prioritizes fairly using vulnerability scores and urgency factors for allocation decisions. It tracks delivery with real-time driver tracking, delivery routes, and last-mile verification using Ghana Card photo, recipient photo, and recipient Ghana Card number verification (signature step removed). It ensures compliance through Ghana Card verification via QoreID (Verify by ID number or image-based workflow), NGO verification by auditors, and donor tax compliance. It provides transparency via audit logs, impact dashboards, and donation certificates. Finally, it supports multiple roles: Admin, Auditor, Donor (Individual, Institution, and Angel), Recipient, NGO, Driver, Field Agent, and Supervisor.

1.3 Scope and Boundaries

The system operates within Ghana's regulatory framework. Donations are denominated in Ghanaian Cedis (GHS). Ghana Card verification is mandatory for most donor types except Angel Donors, who can donate up to GH₵5,000 without ID verification. Corporate donors are subject to the 10% assessable income cap; individual anonymous donations are capped at GHS 10,000. The system does not handle international wire transfers directly; Paystack processes card payments in GHS. The platform is designed for web browsers; mobile apps are out of scope for this document.

1.4 Stakeholder Analysis

Recipients submit aid requests, upload verification documents, and track allocations. Donors (Individual) make monetary or in-kind donations, browse available requests, and view impact. Donors (Institution) have access to the Impact Dashboard, CSR matchmaking, and Find NGOs. Angel Donors donate without Ghana Card verification, with a GH₵5,000 cap per donation. NGOs create projects, manage organization profiles, and receive CSR partnerships. Admins verify users, allocate resources, manage fleet, and view financial reports. Auditors verify NGOs, set audited valuations, and access audit logs. Drivers create trips, update GPS location, and complete delivery with proof. Field Agents upload impact proofs for projects. Supervisors view the delivery dashboard and all trips.

Chapter Two: System Analysis and Specifications

2.1 Executive Summary

ResourceFlow is a full-stack web application with a RESTful Laravel API backend and a React single-page frontend. It supports Hostinger VPS and Laravel Cloud deployments, uses PostgreSQL for data storage, and integrates Paystack for payments in Ghanaian Cedis (GHS). The system includes pre-deployment checks, automated deploy scripts, Nginx configuration, SSL via Let's Encrypt, Supervisor for queue workers, and cron for the Laravel scheduler.

2.2 System Architecture

The system uses a client-server architecture with clear separation of concerns. The frontend is a React SPA served as static files from the Vite build output (frontend/dist). The backend is a Laravel API served via PHP-FPM behind Nginx. The database is PostgreSQL, recommended for production. Payments are handled by Paystack using both webhooks and the frontend SDK. Identity verification is provided by QoreID: Verify by ID number (NIA lookup) and image verification (workflow or manual review when QOREID_WORKFLOW_ID is not configured).

The frontend domain (e.g., yourdomain.com) serves the React app. The API domain (e.g., api.yourdomain.com) serves the Laravel backend. All API calls use VITE_API_BASE_URL, which must end with /api. CORS is configured via SANCTUM_STATEFUL_DOMAINS and CORS_ORIGINS to allow the frontend to make credentialed requests.

2.4 Technical Specifications

The API returns JSON. Rate limiting is set to 60 requests per minute for the API, 10 per minute for Ghana Card verification, and custom limits for uploads. Authentication uses Laravel Sanctum with cookie-based sessions for the SPA. CORS is configured via FRONTEND_URL and CORS_ORIGINS. The password policy enforces a 30-day expiry.

2.5 System Architecture Flow

The browser loads the React SPA from the frontend domain. The SPA makes API calls to the backend domain (VITE_API_BASE_URL). The backend Nginx forwards PHP requests to PHP-FPM, which runs the Laravel application. Laravel connects to PostgreSQL for data. When a donor makes a payment, the frontend redirects to Paystack; on success, Paystack sends a webhook to the backend. The backend verifies the HMAC, creates the Financial record, and marks the donation Verified. Background jobs run via Supervisor (queue:work). Scheduled tasks run via cron (schedule:run). File uploads are stored in storage/app/public and served via the storage symlink.

2.6 Deployment Platform Comparison

Hostinger VPS: full control, Ubuntu 22.04, manual PHP/Nginx/PostgreSQL setup, Certbot for SSL, fixed monthly cost. Setup time 30 to 60 minutes. Laravel Cloud: managed PaaS, automatic PostgreSQL, push-to-deploy, usage-based cost. Setup time under 5 minutes. For budget-conscious deployments with full control, Hostinger is preferred. For speed and automatic scaling, Laravel Cloud is preferred.

3.3 Cardinalities

Cardinality describes how many instances of one entity relate to instances of another. The following relationships are defined:

Users to Requests: One user can create many requests (1:N). Each request belongs to one user.

Users to Donations: One user can make many donations (1:N). Each donation belongs to one user.

Donations to Allocations: One donation can be allocated to many requests through multiple allocation records (1:N). Each allocation links one donation to one request.

Requests to Allocations: One request can receive many allocations from different donations (1:N). Each allocation fulfills one request.

Allocations to Trips: One allocation can have zero or one trip (0..1:1). A trip is created when a driver is assigned to deliver an allocation.

Trips to Delivery Proofs: One trip has exactly one delivery proof when completed (1:1). The delivery proof captures Ghana Card photo, recipient photo, and recipient Ghana Card number verification (signature step removed as of March 2026).

Users to Trips: One user (driver) can perform many trips (1:N). Each trip is performed by one driver.

Organizations to Projects: One organization can manage many projects (1:N). Each project belongs to one organization.

Projects to Contributions: One project can receive many contributions (1:N). Each contribution goes to one project.

Financials: Financial records link to users (mandatory), and optionally to donations or allocations. The relationship is polymorphic in practice: one donation can have one or more financial records (for monetary donations), and one allocation can have financial records for project funding.

3.4 Database Schema Details

The users table stores core identity (name, email, password), role, organization, address, phone, Ghana Card number, verification status (is_verified, verified_at, verified_by), password_changed_at for 30-day expiry, and reputation_score.

The requests table stores aid request details: title, description, need_type, time_sensitivity, recipient_type, region, quantity_required, unit, urgency_score, urgency_level, status (pending, approved, claimed, completed, closed_no_match, cancelled), expires_at, and is_flagged_for_review for heat map stale data indication.

The donations table stores type (Goods, Monetary, Services), item, quantity, remaining_quantity, unit, status, warehouse_id, value, market_price, audited_price, receipt_confirmed_at, and donor compliance fields for tax.

The allocations table links donation_id, request_id, allocated_by, quantity_allocated, status, allocated_date, expected_delivery_date, and actual_delivery_date.

The trips table stores driver_id, request_id, allocation_id, delivery_route_id, status (pending, started, arrived, completed), current_lat, current_lng, started_at, arrived_at, completed_at.

The delivery_proofs table stores trip_id, recipient_photo_path, gha_card_path, recipient_ghana_card_verified, recipient_ghana_card_number, recipient_comments, verified_at, verified_by. The signature_path field is deprecated; completion now requires recipient Ghana Card verification via QoreID.

3.4a Delivery Proof Schema Update (March 2026)

The delivery proof workflow was updated to remove the signature step. Previously, completion required recipient_photo_path, gha_card_path, and signature_path. The new flow requires: (1) Ghana Card photo (upload or capture), (2) Recipient photo (react-webcam), (3) Recipient Ghana Card number entered by the recipient and verified via QoreID. The backend stores recipient_ghana_card_verified (boolean) and recipient_ghana_card_number (string). The signature_path column remains in the schema for backward compatibility but is set to null. Migration 2026_03_08_100000 adds the new columns.

3.5 Migration System and Schema Evolution

The database schema is managed through Laravel migrations. Each migration file defines an up() method for applying changes and a down() method for rolling back. Migrations run in chronological order based on the timestamp in the filename. Key migrations include: 2024_01_12_000001_create_users_table (consolidated user schema), 2024_01_12_000002_create_requests_table (base request structure), 2024_01_12_000004_create_donations_table (donations with warehouse and audit fields), 2024_01_12_000006_create_allocations_table (donation-to-request linking), 2026_02_10_100000_add_heatmap_fields_to_requests_table (region, quantity_required, unit, expires_at), 2026_02_13_200000_create_trips_table (driver tracking), and 2026_02_13_200001_create_delivery_proofs_table (biometric verification). The setup-database.sh script supports --fresh (drop all and recreate), --migrate (run pending only), and --seed-only (re-seed default users).

3.6 Data Integrity and Constraints

Foreign key constraints ensure that allocations reference valid requests and donations, trips reference valid drivers and requests, and delivery proofs reference valid trips. Cascade on delete is used where appropriate: when a request is deleted, its allocations are deleted; when a trip is deleted, its delivery proof is deleted. Nullable foreign keys (e.g., allocation_id in trips) allow optional relationships. The requests table uses a CHECK constraint for status values in PostgreSQL. Indexes on request_id, donation_id, driver_id, and status columns optimize query performance for the heat map, allocation lookups, and trip listings.

Chapter Four: Implementation Guide

4.1 MVC Architecture

ResourceFlow follows the Model-View-Controller (MVC) pattern. On the backend, Laravel provides the structure: Models represent database tables and business logic, Controllers handle HTTP requests and return responses, and the View layer is replaced by JSON API responses consumed by the React frontend.

Models: Each database table has a corresponding Eloquent model (User, Request, Donation, Allocation, Trip, DeliveryProof, Financial, etc.). Models define fillable attributes, casts, and relationships such as belongsTo, hasMany, and hasOne.

Controllers: API controllers live in app/Http/Controllers/Api/. They receive requests, validate input, call models and services, and return JSON. For example, MapController fetches requests and aggregates by region for the heat map. PaymentController verifies Paystack payments and handles the webhook. TripController manages trip creation, location updates, and completion.

Views: The frontend React components act as the view layer. They fetch data from the API, render UI, and send user actions back to the API. There is no server-side templating; the SPA handles all rendering.

4.2 Frontend Component Architecture

The frontend is organized as follows. Components: ProtectedRoute, RoleProtectedRoute, Layout (sidebar by role; angel_donor uses indigo theme), admin (StatCard, AuditReviewModal, etc.), logistics (DriverTripMode, LiveMapTracker, DeliveryVerification), verification (GhanaCardCapture for Ghana Card + selfie), donor (CorporateTaxWidget), ngo (ProjectWizard, NGODashboardWidget), payment (PaymentModal), ui (Button, FileUpload). Pages: admin (AdminDashboard, ResourceAllocation, VerificationCenter, etc.), recipient (RecipientDashboard, CreateRequest), supplier (CreateDonation, AvailableRequests, FindNGOs), ngo (NGODashboard, OrganizationSetup), auditor (NGOVerification, MonetaryTransfers), driver (DriverTripPage, TripsListPage), fieldagent (FieldAgentDashboard, UploadProof). Hooks: useAuth, useAutoRefresh. Services: api.ts (apiClient, requestApi, tripApi, ghanaCardApi, matchmakingApi, etc.). App.jsx defines routes with RoleProtectedRoute; angel_donor is included in donor routes with requiresVerification false for donation creation.

4.3 Backend Middleware

auth:sanctum: API authentication; returns 401 if unauthenticated. password.expired: Checks password_changed_at; if older than 30 days, returns 403 with error_code PASSWORD_EXPIRED; frontend redirects to change-expired-password. verified.user: Restricts request creation and donation creation to verified users; bypassed for role angel_donor (VerifyUserVerified middleware). not.blocked: Blocks users with is_blocked flag. org.tier: Enforces organization tier limits for NGOs. throttle: API 60/min, Ghana Card 10/min, uploads custom. 4.6 Authentication Flow

Login: POST /api/auth/login with email and password. Laravel Sanctum returns a cookie (or token). The frontend stores the session and includes credentials in subsequent requests. Protected routes use auth:sanctum middleware. If the password has expired (password_changed_at older than 30 days), the password.expired middleware redirects to change-expired-password. The verified.user middleware blocks unverified users from creating requests or donations. The RoleProtectedRoute on the frontend restricts pages by role (admin, donor_institution, recipient, etc.).

Protected Routes (Frontend)

ProtectedRoute: Wraps authenticated routes. If the user is not logged in, redirects to /login. If logged in but unverified (is_verified=false) and the role requires verification (recipient, donor_individual, donor_institution, ngo), redirects to /verification-wait. Exception: angel_donor bypasses verification. The verification-wait page renders the DocumentUpload component, allowing users to complete Ghana Card upload or capture later.

RoleProtectedRoute: Restricts access by role. Each route defines allowed roles (e.g., admin, recipient, donor_institution). If the user's role is not in the allowed list, redirects to /dashboard or /verification-wait. Unverified users (except angel_donor) are redirected to /verification-wait before reaching role-specific pages.

Flow: After registration, users who skipped verification are created with is_verified=false. They are redirected to /dashboard; ProtectedRoute intercepts and sends them to /verification-wait. On the Document Upload page they can upload Ghana Card or complete camera capture. Once an admin verifies their documents, is_verified becomes true and they can create requests or donations.

5.1 SVG Heat Map Implementation

The regional request heat map is one of the key deployment features. It visualizes aid request urgency across Ghana's 16 regions using an interactive SVG overlay. The implementation consists of three main parts: the backend MapController that aggregates request data by region, the frontend GhanaSVGHeatMap component that renders the map, and the ghanaMapPaths data file that defines the SVG path coordinates for each region.

5.1.1 Backend: MapController

The MapController serves data at GET /api/requests/map-data. It fetches only verified, active requests (excluding completed, cancelled, closed_no_match) and filters out self-dealing cases where the recipient is also the contributor or assigned supplier. For each request, it computes net need (quantity_required minus quantity already allocated), urgency level (critical, high, medium, low), and urgency_weighted_heat (net need multiplied by a weight: 2.0 for critical, 1.5 for high, 1.0 for medium, 0.5 for low). The response groups data by region and includes total_requests, critical_requests, high_urgency_requests, medium_urgency_requests, low_urgency_requests, urgency_score (a weighted average), net_need, and a list of requests. Admin, Auditor, and Field Agent roles receive high-granularity PII (recipient names); other roles receive aggregated stats only.

// MapController.php - getScopedRequests() filters:
Request::with(['user', 'allocations', 'contributions', 'assignedSupplier'])
    ->whereHas('user', fn ($q) => $q->where('is_verified', true))
    ->whereNotIn('status', ['completed', 'cancelled', 'closed_no_match'])
    ->where(function ($q) {
        $q->whereNull('expires_at')->orWhere('expires_at', '>', now());
    })
    ->get()
    ->filter(function ($request) {
        return !$this->isSelfDealing($request);
    })

5.1.2 Frontend: GhanaSVGHeatMap Component

The GhanaSVGHeatMap component fetches map data from the API every 30 seconds (configurable via updateInterval). It uses a background image (Ghana Map.svg) and an overlay SVG whose paths come from ghanaMapPaths.js. Each region is a path element with a fill color derived from urgency_score: red (#EF4444) for 80+, amber (#F59E0B) for 60+, yellow (#EAB308) for 40+, green (#84CC16) for 20+, blue (#3B82F6) for lower. Regions with flagged or stale data use a gray fill with a dashed stroke. The getHeatColorWithWeight function applies urgency_weighted_heat to scale intensity. On hover, a tooltip shows active requests, net need, total value, urgency level, and nearest warehouse. On click, a modal opens with detailed regional statistics and a quick donate button. The component normalizes region names (e.g., SAVANNAH to Savannah) to match API data with SVG path keys.

// GhanaSVGHeatMap.jsx - Heat color logic
const getHeatColorWithWeight = (urgencyScore, urgencyWeightedHeat = 0) => {
  const baseScore = urgencyScore ?? 0;
  const weighted = urgencyWeightedHeat > 0 ? Math.min(100, baseScore * ...) : baseScore;
  const score = Math.max(baseScore, weighted);
  if (score >= 80) return '#EF4444'; // Red - Critical
  if (score >= 60) return '#F59E0B'; // Amber - High
  if (score >= 40) return '#EAB308'; // Yellow - Medium
  if (score >= 20) return '#84CC16'; // Green - Low-Medium
  return '#3B82F6'; // Blue - Low
};

5.4 Delivery Verification Wizard

The DeliveryVerification component is a four-step wizard. Step 1: Ghana Card capture (file upload or tap to capture). Step 2: Recipient photo (react-webcam getScreenshot; videoConstraints width:640 height:480 facingMode:{ideal:'user'}; onUserMediaError for Chrome compatibility). 5.5 TripController Complete Flow (Code Explanation)

The TripController orchestrates the entire delivery lifecycle. The store method creates a trip with request_id, optional allocation_id and delivery_route_id. Only drivers and admins can create trips. The updateLocation method validates lat/lng (between -90/90 and -180/180), updates current_lat and current_lng, sets status to started if pending, sets started_at on first update, and fires TripLocationUpdated for real-time broadcasting. The arrive method sets status to arrived and arrived_at. The complete method validates recipient_ghana_card_verified (required, must be true) and recipient_ghana_card_number (required). It runs inside a database transaction: creates or updates DeliveryProof with recipient_photo_path, gha_card_path, recipient_ghana_card_verified, recipient_ghana_card_number (signature_path is set to null), marks the trip completed, updates allocation to Delivered, donation to Delivered, request to completed. Authorization: only the assigned driver or admin can complete. The authorizeTripAccess helper allows admin, auditor, the driver, or the request recipient to view the trip.

// TripController.php - complete() transaction logic
return DB::transaction(function () use ($trip, $validated) {
    $proof = DeliveryProof::updateOrCreate(['trip_id' => $trip->id], [...]);
    $trip->update(['status' => 'completed', 'completed_at' => now()]);
    if ($trip->allocation_id) {
        $allocation->update(['status' => 'Delivered', 'actual_delivery_date' => now()]);
        Donation::where('id', $allocation->donation_id)->update(['status' => 'Delivered']);
    }
    $aidRequest->update(['status' => 'completed']);
    return response()->json($trip);
});

Chapter Six: Integrations

6.1 Paystack Payment Integration

Paystack handles all monetary donations in GHS. The integration has three parts: frontend initiation, backend verification, and webhook handling.

Frontend: The donor selects amount and initiates payment. The frontend uses the Paystack inline JS SDK with VITE_PAYSTACK_PUBLIC_KEY. The SDK opens a modal for card entry and redirects to Paystack's hosted page. On success, Paystack redirects back with a reference. The frontend calls POST /api/payments/verify with reference, amount, type, donation_id, and description.

Backend verification: PaymentController::verifyPayment fetches the Paystack API (GET https://api.paystack.co/transaction/verify/{reference}) using the secret key. If the transaction status is success, it creates or retrieves a Financial record with payment_reference, user_id, amount, currency GHS, payment_method paystack, status Completed, and optional donation_id. It uses firstOrCreate to avoid duplicates.

Webhook: Paystack sends charge.success events to POST /api/payments/paystack-webhook. The webhook is public (no auth) but protected by HMAC-SHA512 signature verification. The controller checks for the x-paystack-signature header and computes hash_hmac('sha512', $request->getContent(), $signingSecret). If the signature does not match, it returns 401. This prevents replay attacks and spoofing. On success, it creates a Financial record and, if the metadata includes donation_id and type Donation, marks the Monetary donation as Verified. Duplicate references are ignored.

// PaymentController.php - Webhook signature verification
if (!$request->hasHeader('x-paystack-signature')) {
    return response()->json(['message' => 'Missing signature'], 400);
}
$computedSignature = hash_hmac('sha512', $request->getContent(), $signingSecret);
if ($request->header('x-paystack-signature') !== $computedSignature) {
    return response()->json(['message' => 'Invalid signature'], 401);
}

6.2 QoreID Ghana Card Verification

The GhanaCardVerificationService integrates with QoreID for two verification modes.

6.2.1 Verify by ID Number

POST /api/verify-ghana-card accepts id_number, firstname, lastname, consent_given. The service normalizes input (e.g., gha7000000000 to GHA-700000000-0), validates format with regex /^GHA-\d{9}-\d$/, and calls QoreID at https://api.qoreid.com/v1/gh/identities/ghana-id/{normalized_id} with firstname and lastname. QoreID performs an NIA (National Identification Authority) database lookup. Response: verified (true if ID exists and name matches), name_mismatch (if ID exists but name differs), error_code (NOT_FOUND, NAME_MISMATCH, UNAUTHORIZED, etc.). Rate limit: 10 requests/minute. No workflow or images required.

6.2.2 Image Verification (Ghana Card + Selfie)

POST /api/verify-ghana-card-with-images accepts ghana_card_image/ghana_card_base64, user_photo/user_photo_base64, firstname, lastname, consent_given. Requires QOREID_WORKFLOW_ID. When workflow is not configured: if user is logged in, saves Ghana Card + selfie as VerificationDocument (Pending) for manual review and returns MANUAL_REVIEW; if not logged in (e.g., registration), returns SERVICE_UNAVAILABLE suggesting "Verify by ID number". When configured, calls QoreID workflow submit with documentImage and selfieImage base64.

6.2.3 QoreID Configuration

Environment variables: QOREID_CLIENT_ID, QOREID_SECRET (or legacy QOREID_SECRET_KEY), optional QOREID_WORKFLOW_ID for image verification. QOREID_CAINFO: explicit CA bundle path for Guzzle on Windows (e.g., E:/path/cacert.pem) when php.ini curl.cainfo is not picked up. config/services.php: qoreid.cainfo, qoreid.verify_ssl.

6.2a Angel Donor Flow

Angel Donors register with role angel_donor. No Ghana Card verification is required. The VerifyUserVerified middleware bypasses verification checks for angel_donor. Donations are capped at GH₵5,000 per donation; StoreDonationRequest validates this. CreateDonation.jsx shows the cap message for Angel Donors. Layout sidebar uses indigo theme for angel_donor. RoleProtectedRoute does not redirect Angel Donors to verification. AuthController auto-verifies angel_donor on registration.

6.2d Skip Verification Flow

When camera capture or document upload fails during registration (e.g., CORS errors, 503, camera not accessible on HTTP), users can choose "Continue without verification" to complete onboarding. If the user chooses to skip, Ghana Card upload/capture and NGO RG document are no longer required. Basic fields (name, email, password, role, organization for NGO/institution, etc.) are still required.

Post-registration flow: Users who skip are created with is_verified=false. They are redirected to the dashboard; ProtectedRoute then sends them to /verification-wait. On the Document Upload page they can complete verification later (Ghana Card by ID number, upload, or camera capture when HTTPS is available). Until verification is complete, the backend verified.user middleware blocks them from creating requests or donations. Angel donors are exempt and do not see the skip option.

UI: Register.jsx displays a "Continue without verification — I'll verify later" link in the verification section. When selected, the form allows submit without Ghana Card file or NGO RG document. Success message: "Complete verification from your dashboard to make requests or donations."

6.2b Find NGOs and Matchmaking

GET /api/matchmaking/ngos returns NGOs with projects, completion rate, and impact. Accessible by donor_institution, donor_individual, angel_donor. MatchmakingController::getNGOs fetches organizations with projects; getMatches returns project matches for donors. FindNGOs.jsx displays NGO list, completion badges, projects, impact; CSR partnership modal for corporate donors. NGOSeeder creates 5 NGOs and 10 projects. All NGOs shown (no is_verified filter).

6.2c Camera and react-webcam Integration

GhanaCardCapture: two-step flow—Ghana Card (file or webcam), then selfie. Uses react-webcam with videoConstraints { width: 640, height: 480, facingMode: { ideal: 'environment' } } for Ghana Card, { ideal: 'user' } for selfie. onUserMediaError surfaces permission/device errors; onUserMedia clears error when camera starts. DeliveryVerification: recipient photo via Webcam with same constraints. Chrome compatibility: relaxed facingMode (ideal vs required), explicit width/height, onUserMediaError for feedback when camera fails.

6.3 File Upload Integration

File uploads go through FileUploadController. The upload endpoint accepts a file and type (e.g., delivery_proof, verification, donation). Files are stored in storage/app/public with subdirectories by type. The controller uses rate limiting (throttle:uploads) and returns the stored path. The frontend uses this for verification documents, delivery proofs (Ghana Card, recipient photo, signature), and donation images.

6.4 Optional Pusher Integration

For real-time trip location updates without polling, the system can use Pusher. The backend fires TripLocationUpdated when a driver updates location. Laravel Echo and Pusher on the frontend subscribe to the trip.{id} channel. When BROADCAST_DRIVER=log, events are only logged and the map uses 5-second polling instead.

6.5 DonationComplianceService

The DonationComplianceService enforces donor tax rules. For corporate donors, donations are capped at 10 percent of assessable income. For individual anonymous donations, the limit is GHS 10,000. The service checks these limits before allowing a donation to proceed and stores compliance_agreed when the donor accepts the terms. The corporate tax stats endpoint returns aggregated data for donor impact dashboards.

6.6 API Service Layer (Frontend)

The frontend api.ts (services/api) centralizes API calls. requestApi.getMapData() fetches heat map data. tripApi.updateLocation(), tripApi.arrive(), tripApi.complete() handle trip operations; complete() sends recipient_ghana_card_verified and recipient_ghana_card_number. requestApi.uploadFile() handles file uploads. ghanaCardApi.verify() and ghanaCardApi.verifyWithImages() for Ghana Card. matchmakingApi.getNGOs() for Find NGOs. Axios is configured with baseURL from VITE_API_BASE_URL (fallback http://127.0.0.1:8000/api), withCredentials: true for Sanctum cookies, and interceptors for 401/403 (password expired redirect). TanStack React Query provides caching, refetching, and loading states.

6.7 Paystack Payment Flow (Detailed)

Donor selects amount on CreateDonation. Frontend initializes Paystack with VITE_PAYSTACK_PUBLIC_KEY. Paystack.Inline() opens modal; donor enters card. On success, Paystack returns reference. Frontend calls POST /api/payments/verify with reference, amount, type=Donation, donation_id, description. Backend PaymentController::verifyPayment fetches GET https://api.paystack.co/transaction/verify/{reference} with secret key. If status success, creates Financial record (payment_reference, user_id, amount, currency GHS, status Completed, donation_id). Updates donation status to Verified. Alternatively, Paystack webhook POST /api/payments/paystack-webhook sends charge.success; backend verifies HMAC-SHA512 (x-paystack-signature), creates Financial, updates donation. Webhook is primary for reliability (handles closed-tab scenarios).

6.8 Vulnerability Scoring and Urgency

Requests have urgency_score (0-100) and urgency_level (critical, high, medium, low). UrgencyCalculationService computes based on time_sensitivity, recipient_type, need_type, and other factors. VulnerabilityScore model stores per-user scores. MapController uses urgency_weighted_heat = net_need * weight (2.0 critical, 1.5 high, 1.0 medium, 0.5 low) for heat map coloring. AllocationController::getPrioritizedRequests returns requests ordered by vulnerability and urgency for admin allocation decisions.

Chapter Six A: Risk Assessment and Production Recommendations

6A.1 Risk Assessment

Unauthorized financial creation is a critical risk: any authenticated user could theoretically create Financial records without real payment. Mitigation: restrict Financial CRUD to admin and auditor; require payment_reference for Completed status. Admin bypassing verification is high risk: an admin could mark unverified users as verified. Mitigation: audit trail for verification actions. Paystack webhook spoofing is medium risk; HMAC-SHA512 verification is implemented. CORS misconfiguration is medium risk; SANCTUM_STATEFUL_DOMAINS and CORS_ORIGINS must be set correctly. Password expiry lockout is low risk; the change password flow and admin reset handle it.

6A.2 Production Deployment Checklist

Before deployment: Run ./scripts/pre-deploy-check.sh. Set APP_ENV=production, APP_DEBUG=false. Configure DB_* credentials. Set Paystack live keys (PAYSTACK_PUBLIC_KEY, PAYSTACK_SECRET_KEY). Set SANCTUM_STATEFUL_DOMAINS to include frontend domain (e.g., yourdomain.com). Set CORS_ORIGINS. Configure QoreID: QOREID_CLIENT_ID, QOREID_SECRET; optional QOREID_WORKFLOW_ID for image verification; QOREID_CAINFO if on Windows. Set FRONTEND_URL and APP_URL.

After deployment: Change Super Admin and Auditor passwords immediately. Configure Paystack webhook URL (https://api.yourdomain.com/api/payments/paystack-webhook). Test monetary donation flow. Verify queue worker (Supervisor) runs queue:work if queues used. Verify cron for schedule:run. Test file upload and storage:link. Test Ghana Card Verify by ID number. Test Angel Donor registration and donation cap. Test Find NGOs. Test delivery verification (Ghana Card, recipient photo, recipient Ghana Card verify).

Health checks: curl https://api.yourdomain.com/api/health (if implemented). Verify frontend loads. Run php artisan migrate:status. Test Super Admin login. Verify CORS (frontend can call API). Check storage permissions.

6A.3 QoreID Production Configuration

For full image verification: Create a workflow in QoreID dashboard (https://dashboard.qoreid.com). Copy the workflow ID. Add QOREID_WORKFLOW_ID=your_workflow_id to .env. Without it, image verification saves for manual review (logged-in users) or suggests Verify by ID number (registration). For Windows servers with Guzzle SSL issues: Download cacert.pem from curl.se, set QOREID_CAINFO to full path. Verify QOREID_CLIENT_ID and QOREID_SECRET are set for token auth.

6A.4 Deployment Challenges

Common issues encountered when deploying to VPS (e.g., Hostinger):

Git setup and ownership: If the VPS repo was not cloned via git, initialize with git init and add remote. "Dubious ownership" errors occur when the repo is owned by a different user; resolve with git config --global --add safe.directory /path/to/repo.

Divergent branches: VPS may have local commits that conflict with origin. To match GitHub exactly: git fetch origin && git reset --hard origin/main.

CORS and API URL: Frontend must call the correct API base URL. If VITE_API_BASE_URL is missing or points to 127.0.0.1:8000, CORS errors occur. Create frontend/.env with VITE_API_BASE_URL=http://your-vps-ip:8080/api (or https://api.yourdomain.com/api when SSL is configured). Rebuild with npm run build. Backend .env must set APP_URL, FRONTEND_URL, SANCTUM_STATEFUL_DOMAINS, and CORS_ORIGINS to include the frontend origin.

Nginx ports: Frontend typically on port 80; API on a separate port (e.g., 8080) or subdomain. Ensure Nginx server blocks route /api to the Laravel backend and static assets to frontend/dist.

Camera and HTTPS: Browser getUserMedia (camera) is blocked on HTTP except localhost. On production without SSL, users cannot capture Ghana Card or selfie; they must use file upload or skip verification. Install Certbot (apt install certbot python3-certbot-nginx) and run certbot --nginx when ready for HTTPS.

Storage and permissions: Ensure storage and bootstrap/cache are writable (chmod -R 775). Run php artisan storage:link. Create backend/resources/views if view:cache fails.

Appendix D: Troubleshooting Guide

500 Server Error: Check backend storage/logs/laravel.log, verify .env exists and permissions on storage and bootstrap/cache (chmod 775). 403 Forbidden: Ensure Nginx root points to backend/public. 502 Bad Gateway: Restart PHP-FPM (systemctl restart php8.2-fpm). CORS errors: Add frontend domain to SANCTUM_STATEFUL_DOMAINS and CORS_ORIGINS; ensure VITE_API_BASE_URL in frontend/.env points to the correct API URL; rebuild frontend. Paystack webhook fails: Verify URL is https://api.yourdomain.com/api/payments/paystack-webhook, check route exists, verify HMAC secret. Login fails after 30 days: Password expired; use change-expired-password. Database connection refused: Check PostgreSQL is running, verify DB_* in .env, check pg_hba.conf. Frontend blank page: Verify VITE_API_BASE_URL ends with /api, rebuild with npm run build. Assets not loading: Run npm ci && npm run build. Git dubious ownership: git config --global --add safe.directory /path/to/repo. Divergent branches on VPS: git fetch origin && git reset --hard origin/main. QoreID cURL error 60 (SSL): Set QOREID_CAINFO to full path to cacert.pem; ensure file exists. Camera not activating in Chrome: Check site has camera permission; use HTTPS or localhost (camera blocked on HTTP in production); ensure videoConstraints use facingMode: { ideal: 'user' } not strict 'user'; onUserMediaError will show permission/device errors. Image verification "workflow not configured": Set QOREID_WORKFLOW_ID in QoreID dashboard and .env, or use Verify by ID number.

Appendix E: Delivery Proof Completion Payload

POST /api/trips/{id}/complete accepts: gha_card_path (nullable), recipient_photo_path (nullable), recipient_ghana_card_verified (required, boolean, must be true), recipient_ghana_card_number (required, string), recipient_comments (optional). The backend validates recipient_ghana_card_verified.accepted and recipient_ghana_card_number.required. signature_path is no longer accepted; the field is set to null in DeliveryProof. This ensures the recipient has verified their Ghana Card via QoreID before delivery can be completed.

Appendix F: Ghana Card Verification Flow Decision Tree

Registration: If angel_donor → skip verification. Else: Option A) Verify by ID number (POST /verify-ghana-card) → instant. Option B) Image verification (Ghana Card + selfie): If QOREID_WORKFLOW_ID set → QoreID workflow. Else if logged in → save for manual review. Else → suggest Verify by ID number. Option C) Skip verification: User chooses "Continue without verification"; registers with is_verified=false; redirected to /verification-wait to complete later. Delivery: Recipient must enter Ghana Card number and verify via Option A before driver can complete.

Appendix K: Glossary

Angel Donor: Donor type with no Ghana Card verification; donations capped at GH₵5,000. NIA: National Identification Authority (Ghana). QoreID: Third-party service for Ghana Card verification. Verify by ID number: QoreID NIA lookup using Ghana Card number + name. Image verification: Ghana Card photo + selfie submitted to QoreID workflow. recipient_ghana_card_verified: Boolean indicating recipient verified their Ghana Card at delivery. recipient_ghana_card_number: Ghana Card number entered and verified by recipient. StoreDonationRequest: Laravel form request validating donation creation; enforces Angel Donor cap. GhanaCardCapture: React component for Ghana Card + selfie capture. DeliveryVerification: Four-step wizard for delivery proof (Ghana Card, recipient photo, recipient verify, submit).

Appendix L: Request Flow for Delivery Completion

1. Driver creates trip (POST /api/trips). 2. Driver starts trip, DriverTripMode begins watchPosition, sends PUT /api/trips/{id}/update-location every 15s. 3. Driver arrives, POST /api/trips/{id}/arrive. 4. Driver opens DeliveryVerification. Step 1: Upload or capture Ghana Card image, POST /api/files/upload, store path. Step 2: Capture recipient photo via Webcam getScreenshot, upload, store path. Step 3: Hand device to recipient. Recipient enters Ghana Card number. Driver clicks Verify. Frontend calls POST /api/verify-ghana-card with id_number, firstname, lastname. Backend GhanaCardVerificationService calls QoreID NIA lookup. If verified, set recipient_ghana_card_verified=true, recipient_ghana_card_number. Step 4: Driver clicks Complete. POST /api/trips/{id}/complete with gha_card_path, recipient_photo_path, recipient_ghana_card_verified, recipient_ghana_card_number. Backend creates DeliveryProof, updates trip, allocation, donation, request. 5. Allocation status=Delivered, Donation status=Delivered, Request status=completed.
